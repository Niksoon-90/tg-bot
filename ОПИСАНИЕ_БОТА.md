# Описание работы бота отчётов по таблице Google Sheets

## Назначение

Бот загружает таблицу из Google Sheets по расписанию и по запросу пользователя, сравнивает данные и показывает **изменения в столбце «Количество ресурсов, необходимое к подбору»** в формате:

**Тип ТК - ТК (Город, тип ресурса) было → стало**

Строка в таблице определяется парой: **ТК** + **Тип ресурса** (нормализация: обрезка пробелов и схлопывание пробелов внутри строки).

---

## Хранимые снимки (JSON, папка `sheet_versions/`)

| Файл | Когда создаётся | Назначение |
|------|------------------|------------|
| **sheet_day_start_YYYYMMDD.json** | 00:30 МСК каждый день (при ошибке — повтор каждые 10 мин до ~05:00). Либо при первом запросе пользователя за день, если снимка ещё не было. | «Старт дня» — с чем сравнивать отчёт «за день». |
| **sheet_day_end_YYYYMMDD.json** | 23:00–23:10 МСК каждый день (один запрос, сохраняется снимок). | Конец дня — для утреннего отчёта «с последнего запроса до конца вчера». |
| **sheet_last_user_request.json** | Только когда пользователь сам нажимает /update (или /report). | «Текущие данные» — база для кнопки «С последнего запроса» и для блока «Изменения с последнего запроса» в рассылке. |

Дополнительно: `reports/last_fetch_time.txt` — время последней успешной загрузки таблицы; `reports/last_morning_report.txt` — дата последней отправки утреннего отчёта (чтобы не слать дважды за день).

---

## Расписание (фоновая задача каждые 10 минут, время МСК)

1. **00:30–05:00**  
   Если снимка на начало сегодняшнего дня ещё нет — запрос таблицы и сохранение в `sheet_day_start_YYYYMMDD.json`. При ошибке повтор через 10 мин.

2. **23:00–23:10**  
   Если снимка на конец сегодняшнего дня ещё нет — запрос таблицы и сохранение в `sheet_day_end_YYYYMMDD.json`.

3. **09:10–09:20, пн–пт**  
   Утренний отчёт подписчикам: «С момента последнего вашего запроса до конца дня [вчера] были внесены следующие изменения» — сравнение `sheet_last_user_request.json` и `sheet_day_end_YYYYMMDD.json` за **вчера**. Отправляется один раз в день (флаг в `last_morning_report.txt`).

4. **9:30–17:30, пн–пт, каждые 30 минут**  
   Рассылка подписчикам: один запрос к таблице Google, затем два блока в одном сообщении:
   - **За сегодня:** список изменений (текущая таблица vs `sheet_day_start_YYYYMMDD.json`) в формате «Тип ТК - ТК (Город, тип ресурса) было → стало».
   - Ниже разделитель и блок **Изменения с последнего запроса:** список изменений (текущая таблица vs `sheet_last_user_request.json`) в том же формате.

   **Пример сообщения рассылки:**

   ```
   📋 Рассылка:

   За сегодня:

     ГМ - 177 (Ижевск, Авто курьер) 6 → 4
     ПВЗ - 42 (Москва, Пеший курьер) 2 → 3

   ——————————————
   Изменения с последнего запроса:

     ПВЗ - 42 (Москва, Пеший курьер) 2 → 3
   ```

   Если изменений нет в каком-то блоке — выводится «Изменений нет.» в этом блоке.

## Действия пользователя

### /update (или /report)

1. Запрос текущей таблицы из Google Sheets.
2. Загрузка снимка на начало дня за сегодня (`sheet_day_start_YYYYMMDD.json`).
3. Если снимка нет — **текущая выгрузка сохраняется как снимок дня**, затем сравнение идёт с ней же (изменений за день будет 0).
4. Сравнение: текущая таблица vs снимок на начало дня.
5. В чат выводится отчёт: **«Изменения за сегодня (Количество ресурсов, необходимое к подбору):»** — список строк в формате «Тип ТК - ТК (Город, тип ресурса) было → стало», при отсутствии изменений — «Изменений за сегодня нет.»; при наличии добавленных/удалённых строк — отдельный блок.
6. Текущая выгрузка сохраняется в **sheet_last_user_request.json** (это и есть «последний запрос пользователя»).

Если загрузка таблицы или снимка начала дня недоступна — пользователю показывается сообщение об ошибке или «Нет снимка на начало дня…» (в текущей логике снимок при первом запросе создаётся из этого же запроса, так что этот кейс редкий).

---

### Кнопка «С последнего запроса»

- Запрос текущей таблицы из Google Sheets.
- Сравнение с **sheet_last_user_request.json** (последний /update).
- В чат выводится: **«Изменения с последнего запроса (дата/время):»** и список изменений в формате «Тип ТК - ТК (Город, тип ресурса) было → стало».
- **sheet_last_user_request при нажатии кнопки не обновляется** — обновление только по /update.

Если нет сохранённого запроса — сообщение: «Нет сохранённых данных последнего запроса. Сделайте /update…».

---

### Кнопка «Посмотреть последние данные»

- Показывает время последней успешной загрузки таблицы из Google Sheets (из `last_fetch_time.txt` или по времени сохранённых снимков/отчётов).

---

### /history, /history_today, /history_yesterday

- История изменений по столбцу «Количество ресурсов, необходимое к подбору» за выбранный период.
- Данные берутся из файлов **changes_*.json** в `reports/` (они создаются при каждом успешном /update с ненулевым diff).
- Формат вывода: период и список строк «Тип ТК - ТК (Город, тип ресурса) было → стало».

---

### /subscribe и /unsubscribe

- **/subscribe** — добавление чата в рассылку. Подписчики получают:
  - **Утренний отчёт** (пн–пт 09:10 МСК): изменения с последнего запроса пользователя до конца вчерашнего дня.
  - **Рассылка каждые 30 минут** (пн–пт 9:30–17:30 МСК): блок «За сегодня» и блок «Изменения с последнего запроса» (см. пример выше).
- **/unsubscribe** — отписка от рассылки.

---

### /last_update

- Текст с датой/временем последней загрузки данных из Google Sheets (то же, что кнопка «Посмотреть последние данные»).

---

## Важные детали

- **Снимок на начало дня** создаётся в 00:30 МСК с повтором при ошибке каждые 10 мин; если пользователь первый раз за день запрашивает отчёт до 00:30 (или снимок по какой-то причине не создался), то **его запрос сохраняется как снимок дня**, и отчёт «за день» строится уже от него.
- **«Текущие данные»** обновляются **только по /update**, не по расписанию и не по кнопке «С последнего запроса».
- Сравнение строк идёт по ключу **(ТК, Тип ресурса)** с нормализацией пробелов, чтобы не дублировать одинаковые по смыслу строки.
- Старые снимки и отчёты удаляются по давности (RETENTION_DAYS, порядка 183 дней).

## Краткая схема

```
00:30 МСК (или первый /update за день) → sheet_day_start_YYYYMMDD.json
23:00 МСК                              → sheet_day_end_YYYYMMDD.json
/update пользователя                   → отчёт «за день» + sheet_last_user_request.json
Кнопка «С последнего запроса»          → текущая таблица vs sheet_last_user_request (без обновления last)
09:10 пн–пт (подписчики)               → утренний отчёт: sheet_last_user_request vs sheet_day_end_вчера
9:30–17:30 пн–пт каждые 30 мин         → рассылка: «За сегодня» + «Изменения с последнего запроса»
```

После правок в этом описании можно скорректировать код под вашу желаемую логику.

## Деплой: как сохранить JSON при редеплое

При редеплое (Docker, Heroku, Railway и т.п.) файловая система контейнера обычно создаётся заново, и каталоги `sheet_versions/` и `reports/` очищаются — снимки и отчёты теряются.

**Что сделать:** вынести данные на постоянный том и указать его через переменную окружения:

- **Переменная:** `LIKA_DATA_DIR` — корневая папка для данных. В ней создаются подпапки `sheet_versions` и `reports`.

**Примеры:**

1. **Docker** — смонтировать volume и передать путь:
   ```bash
   docker run -e LIKA_DATA_DIR=/data -v lika_data:/data ...
   ```

2. **Docker Compose** — volume для сервиса бота:
   ```yaml
   environment:
     - LIKA_DATA_DIR=/data
   volumes:
     - lika_data:/data
   ```

3. **VPS / свой сервер** — указать каталог на диске, который не пересоздаётся при деплое:
   ```bash
   export LIKA_DATA_DIR=/var/lib/lika
   ```

Если `LIKA_DATA_DIR` не задана, используется каталог рядом с кодом (как раньше); для локального запуска ничего менять не нужно.
